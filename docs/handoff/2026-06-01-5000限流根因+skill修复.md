## 根因定位（最关键，务必读懂）

### 真实根因：飞书 base/v3 写限流，伪装为 `code=5000 msg=空`

经过系统排查排除了以下假设：

| 假设 | 排除依据 |
|---|---|
| User Token 失效 | `auth status -o json` 显示 healthy，base 全套 scope |
| base 权限不足 | 间隔写时单字段 upsert 成功 |
| 软段文本非法字符 | 4 个软段单独 upsert 成功 |
| select 选项值不存在 | 7 个 select 字段单独 upsert 成功 |
| 字段数/payload 上限 | **单字段** upsert 在密集请求后也 5000 |

**真相**：密集连续调 `feishu-cli bitable record upsert`（诊断过程中共发了 10+ 次写请求），飞书对同一 base 触发了限流/风控，此后**读写全部报 `code=5000 msg=空`**，持续约数分钟。

**验证方式**：同样内容在间隔 30 秒后重跑，之前失败的请求恢复成功。

### 错误方向（已回退）

基于"字段数上限"假设，给 `sync.py` 加了"全量失败 → 递归二分降级"逻辑，但这在限流场景下**反效果**（连发更多请求加剧限流）。已完整回退，sync.py 代码与 main 分支一致（无残留改动）。

### 独立已知问题（低优先级）

`parent_project` 那张关联表（`tblZ5Zu8v6m5AUx0`）调 `record list --limit 200` 也报 5000。
这是 sync.py 的 `parent_project` link 同步路径，失败会 print warning 并跳过，不阻断主流程。
疑似该表的 limit=200 超了某个阈值，或 token scope 问题。暂未深查。
