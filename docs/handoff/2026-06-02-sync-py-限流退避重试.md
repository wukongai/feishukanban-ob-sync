# Handoff：sync.py 加指数退避重试 — 应对飞书 base/v3 写限流

**日期**：2026-06-02  
**优先级**：P2（偶发限流，用户无感自动重试）  
**版本目标**：v0.7.4（patch）

---

## 背景

飞书多维表格 base/v3 API 在密集连续写请求后会触发限流，表现为：

```
base/v3 API 失败: code=5000, msg=（空）
```

- `code=5000 + msg 为空` = 限流特征（有 msg 内容 = 真实业务错误，不要重试）
- 间隔 30 秒后同样请求恢复成功（验证来自 2026-06-02 真实事故）
- 根因分析详见 `docs/handoff/2026-06-01-5000限流根因+skill修复.md`

当前 sync.py 没有任何重试机制，限流直接报错退出，用户需要手动重跑。

---

## 修复方案：指数退避重试

在 CLI 调用层加重试——遇到 `code=5000 + msg 空` 最多重试 3 次，等待间隔 5s / 15s / 45s。

### 改动位置

**`run_cli` 函数**（sync.py 约 L1409 附近）或 **`feishu_upsert_record`**（约 L2101）。

推荐改 `run_cli`，因为所有飞书 CLI 调用都经过这里，改一处全局生效。

### 重试条件（严格）

```python
def _is_ratelimit_5000(stderr: str, stdout: str) -> bool:
    """
    飞书限流特征：code=5000 且 msg 为空。
    有 msg 内容 = 真实业务错误（字段非法、权限不足等），不重试。
    """
    return "code=5000" in stderr and 'msg=' in stderr and 'msg=\n' in stderr
```

### 伪代码

```python
import time

MAX_RETRY = 3
BACKOFF = [5, 15, 45]  # 秒

def run_cli(cmd: list[str], ...) -> subprocess.CompletedProcess:
    for attempt in range(MAX_RETRY + 1):
        result = subprocess.run(cmd, ...)
        if result.returncode == 0:
            return result
        if _is_ratelimit_5000(result.stderr, result.stdout) and attempt < MAX_RETRY:
            wait = BACKOFF[attempt]
            print(f"  ⏳ 飞书限流(code=5000)，{wait}s 后重试（{attempt+1}/{MAX_RETRY}）...")
            time.sleep(wait)
            continue
        return result  # 真实错误或已超最大重试，直接返回让上层处理
    return result
```

### 关键约束

- **只重试 `code=5000 + msg 空`**，其他错误码立即返回，不重试
- **最多 3 次**，总最长等待 65 秒（5+15+45），超过则返回失败让用户知道
- **print 进度**：每次重试都输出提示，用户知道在等而不是卡死
- **不改 dry-run 行为**：dry-run 不调 CLI，无需改动
