# 8 条架构设计原则

> 来源:全局 `~/.claude/CLAUDE.md`「架构设计核心原则」。本文是项目级展开版,加 feishukanban-ob-sync 特定案例。

任何架构设计 → 开工前对照 + 完成后自评。任一条违反 → 暂停 → 重新设计或显式说明「为什么本次违反 + 未来如何修复」。

---

## 8 条原则(按优先级)

| # | 原则 | 自检问题 |
|---|------|----------|
| 1 | **解耦** | 各组件能否独立修改?某个改动会引发多少处连锁修改?依赖方向是否清晰单向? |
| 2 | **可扩展** | 加新类型 / 新功能要不要改核心代码?还是只加新文件 / 新配置?是否预留了扩展点? |
| 3 | **灵活修改** | 局部修改的成本与影响半径多大?回滚是否容易?修改是否需要全局重测? |
| 4 | **渐进披露** | 新人看是从最简单进入还是被一堆细节淹没?复杂度是否分层暴露? |
| 5 | **鲁棒性** | 边界情况 / 错误输入 / 异常状态都有兜底吗?失败时是否优雅降级?关键路径有 fallback 吗? |
| 6 | **人可读 + 可教学** | 命名 / 结构 / 文档是否让没参与的人能读懂?能否被讲清楚教给别人? |
| 7 | **高复用 + 易移植** | 跨项目 / 跨语言能用吗?依赖是否最小化?业务逻辑和基础设施是否分离? |
| 8 | **工程化清晰** | 目录结构 / 命名 / 文档是否符合项目规范?流水线是否可自动化? |

**冲突取舍**:按优先级顺序(解耦 > 可扩展 > 灵活修改 > 渐进披露 > 鲁棒性 > 人可读 > 高复用 > 工程化)。冲突时**显式记录权衡过程**。

---

## 强制执行流程

### 设计前(动手写代码 / 文档之前)

对照 8 条做「反向打分」,列出每条的**具体落地表现**。

**示例**(v0.3.7 反向字段 diff sync 设计前):

> 解耦 → "新加 3 个 helper(`_extract_fields_from_feishu_row` / `_strip_wikilink` / `_diff_frontmatter_with_feishu`)独立,可单独测试;diff 计算与 apply 分离"
> 可扩展 → "新同步字段加到 `_REVERSE_SYNC_FIELD_WHITELIST` 常量一行,特殊处理在 helper elif 分支加"
> 鲁棒性 → "飞书空 + OB 有值 → 保留 OB(防误清);parent_project 暂禁(规避 v0.2.5 bug 放大)"

### 完成后(交付 / commit 之前)

再过一遍 8 条做「自评报告」附 CHANGELOG 该版本 entry。

**示例**(v0.3.7 CHANGELOG entry 末尾):

```markdown
### ⚖️ 8 条原则自评

| # | 原则 | 评分 | 备注 |
|---|---|---|---|
| 1 | 解耦 | ⭐⭐⭐⭐⭐ | 3 个 helper 独立,可单独测;diff 计算与 apply 分离 |
| 2 | 可扩展 | ⭐⭐⭐⭐⭐ | 加新同步字段 → 加白名单一行 |
| 5 | 鲁棒性 | ⭐⭐⭐⭐⭐ | 飞书空保留 OB(防误清);parent_project 暂禁(避免 v0.2.5 bug 放大) |
| ... |
```

### 违反原则硬规则

- ❌ 不许「事后补」 — 开工前就识别
- ✅ 显式在交付物里写 **「本次违反 X 原则,原因:____,未来修复路径:____」**
- ✅ 计入项目技术债清单(写到 [`../ROADMAP.md`](../ROADMAP.md) 或单开 `tech-debt.md`)

## feishukanban-ob-sync 已知违反 / 技术债

### 违反 ⑦「高复用」:sync.py 单文件膨胀

- **现状**:sync.py 3300+ 行,业务 + 飞书 API + OB 操作 + CLI argparse 混在一起
- **原因**:早期开发节奏快,文件膨胀比拆分有效率
- **未来修复路径**:v0.5+ 考虑拆 `lib/` 子模块(`feishu_api.py` / `ob_io.py` / `models.py` / `cli.py`)— 等核心功能稳定后再做,避免 over-engineering
- **缓解**:`docs/ARCHITECTURE.md` 维护好「函数 → 行号 → 职责」索引

### 违反 ⑤「鲁棒性」:install.sh `sed -i ''` macOS 专属语法

- **现状**:Linux 用户跑会报错(BSD sed 与 GNU sed 不兼容)
- **未来修复路径**:抽 `sed_inplace()` helper,detect OS 后选 `sed -i ''` 或 `sed -i`
- **影响**:阻塞了 v0.4 跨平台目标

### 违反 ⑧「工程化」:0 自动化测试

- **现状**:全部靠真实 vault 手动验证(铁律 #6)
- **原因**:跑通比 robust 更优先(早期开发期)
- **未来修复路径**:v0.5+ 加 pytest 单元测试(覆盖 helper 纯函数为主,集成测试用 mock 飞书 API)
